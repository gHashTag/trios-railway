## Follow-up note (non-blocking)

`ssot.janitor_status` view uses an inline subquery against `pg_locks` for the
lock holder. When no advisory lock is held, the subquery returns zero rows and
the entire view yields zero rows rather than a single row with NULL lock fields
+ counts. The COMMENT in 0013 says "NULL when lock is free", so this is a minor
mismatch. Fix would be a follow-up migration converting the subquery to a
LEFT JOIN against a singleton row. Not patched here to avoid unsolicited DB
mutation; recommended as a small successor to L-SS5.
